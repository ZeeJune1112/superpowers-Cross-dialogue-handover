# Skill: Handoff Protocol (状态交接协议)

> **Version**: 1.2
> **配合协议**: 需与 `Handoff Protocol (知识固化)` 配合使用。本 Skill 负责动态状态交接，HANDOFF_PROTOCOL.md 负责确定性知识归档。

## 触发条件

当用户在对话中输入指令 `/handoff`，或**明确要求立即停止并**"执行交接"、"上下文满了"时，必须立即中断当前正在进行的其他代码生成或分析任务，仅执行此交接协议。
**（注意：若用户表示"先做完最后一步再交接"，则必须等待当前任务结束后再触发，防止假触发打断工作流。）**

## 前置检查：HANDOFF_PROTOCOL 是否已执行

在执行交接前，检查是否有"已完成但未归档"的代码工作：

- 列出 `docs/superpowers/architecture/features/` 和 `docs/superpowers/architecture/pitfalls/` 中的 .md 文件
- 对比 `docs/superpowers/architecture/INDEX.md`（若存在）的「已完成功能」和「已知坑点」表格：
  - 若 features/ 或 pitfalls/ 目录中有 .md 文件，但 INDEX.md 中缺少对应条目 → HANDOFF_PROTOCOL **尚未完成**
  - 此时**拒绝执行 /handoff**，告知用户："记忆固化尚未完成。请允许我先执行 HANDOFF_PROTOCOL 进行知识归档，归档完成后再执行 /handoff。"
  - 然后立即执行 HANDOFF_PROTOCOL 的 5 个步骤，完成后再继续本协议的后续步骤
- **不检查 specs/ 和 plans/**：纯设计/规划窗口无需 HANDOFF_PROTOCOL，直接 /handoff 即可
- 若 INDEX.md 不存在且 features/ 和 pitfalls/ 也均为空 → 无代码产出需归档，可直接继续

## 执行动作 (严格按序执行)

一旦触发，你必须将当前工作区的瞬时状态持久化到项目规划文件中。请严格按以下步骤操作：

### 1. 状态盘点

梳理当前对话中已成功应用的代码、尚未完成的功能以及任何未解决的 Bug/报错。

**（注意：严格禁止在后续写入中粘贴大段代码，仅使用精简的自然语言描述逻辑变更、类名或方法名。）**

### 2. 更新 Plan

根据上下文锁定对应的 plan 文件。plan 文件存放在 `docs/superpowers/plans` 目录下（相对于项目根目录）。

**路径解析规则**：若当前处于 git worktree 中，以 worktree 的根目录为基准扫描 `docs/superpowers/plans`；否则以当前工作目录为基准。写入时在 Handoff Context 中标注实际写入路径，便于 continue-handoff 跨目录定位。

**多文件匹配规则（按优先级依次尝试）**：

- 用户在当前对话中**最近明确指定或引用**过的 plan 路径（如 `@docs/superpowers/plans/xxx.md`）；
- 用户在当前对话中**要求 AI 读取或写入**过的 plan 文件；
- 若无上述记录，则扫描 `docs/superpowers/plans` 目录：
  - 若目录下只有一个 `.md` 文件 → 直接使用；
  - 若目录下有多个 `.md` 文件 → **优先选择最近修改时间最新的文件**（与 /continue-handoff 的选取策略一致），在交接报告中标注所选文件路径。若选取错误，用户可在下个窗口手动指定；
  - 若目录下没有任何 `.md` 文件 → 在 `docs/superpowers/plans` 下新建 `plan.md`，并向用户说明"未检测到现有 plan，已创建默认 plan 文件"。

将其中已经确实完成的子任务标记为完成 `[X]`。

**TodoWrite 同步（若适用）**：若当前对话中使用了 TodoWrite 跟踪任务进度，以 TodoWrite 的完成状态为准更新 plan 文件的 checkbox。TodoWrite 是运行时 source of truth，plan 文件的 checkbox 是持久化镜像。

**任务完成判定标准（满足任一即可标记为 `[X]`）**：

1. TodoWrite 中该任务状态为 `completed`；
2. 用户明确确认该步骤已完成；
3. 该步骤对应的代码/配置已实际写入文件并通过基础验证（有 Write/Edit 工具调用记录）；
4. 该步骤为纯文档/规划类任务，且内容已在对话中定稿。

**若不确定是否完成，保持 `[ ]` 不变**，并在 Handoff Context 的 `Pending Queue` 中说明。

**任务行格式兼容**：行样式为 `- [ ] **Step N: 任务描述**` 或 `- [X] **Step N: 任务描述**`。当确认某任务已完成后，将 `[ ]` 修改为 `[X]`，保留加粗和描述部分不变。若某行不是以上述格式开头，则不自动修改，仅在交接节点中文本说明完成状态。

**若 plan.md 文件格式不兼容（不使用标准任务行格式）**：仅追加 Handoff Context，不自动修改任务状态，并在 Handoff Context 中备注"Plan 格式非标准，任务状态未自动更新"。

### 3. 注入交接节点

在 `plan.md` 文件的最底部，定位 `## Handoff Context (交接上下文)` 节点。

- 如果已存在，**保留最近 3 次历史交接记录**，删除编号最小（最旧）的记录，然后追加新的记录；
- 如果不存在则强制创建。

**记录格式**：每次交接记录必须包含编号和时间戳作为分隔标记，格式如下：

```markdown
---
### Record #N — YYYY-MM-DD HH:MM
- **Plan Path**: [实际写入的 plan 文件完整路径]
- **Git Branch**: [当前分支名]
```

编号从 1 开始递增。每次追加新记录时，若已有 3 条记录，先删除 `#` 编号最小的那条，再追加 `#` 编号为 最大编号+1 的新记录。

> **设计意图**：保留最近 3 次交接记录，便于新窗口追溯历史上下文，同时避免 plan.md 过度膨胀。若任务跨 4 个以上窗口，最早记录的历史细节可从 git log 和 feature 归档中恢复，不会丢失关键信息。

### 4. 结构化写入

在交接节点下，必须包含以下五个维度（保持极简和精准）：

- **Current Focus (当前焦点)**: 一句话说明我们目前停在哪个功能节点。
- **Last Action (上一步动作)**: 最后修改或测试了哪些具体的脚本、配置文件或场景设置。**（注意：必须明确指出哪些代码是"已经写入文件"的，哪些是"仅讨论过方案但未写入"的）**。状态标注规则如下（仅以实际工具调用为准，每项择一使用）：
  - `(Written)` — 已写入文件，尚未验证
  - `(Verified)` — 已写入并通过基础验证（运行/测试通过）
  - `(Committed)` — 已 commit 到版本库
  - 仅在对话中讨论过方案但无 Write/Edit 调用时，不标注状态，描述为"仅讨论，未写入"
- **Architecture & Dependencies (架构与依赖状态)**: 记录当前阶段是否引入了新的依赖、修改了 IoC 容器的注册逻辑，或定下了特定的代码规范（如特定的异步处理方式）。若无则填"无"。
- **Blockers & Logs (阻碍与报错)**: 如果带着 Bug 结束，附上关键的报错堆栈、失败原因或已尝试但无效的方案。区分阻碍类型：
  - `(Blocking)` — 必须解决后才能继续
  - `(Non-blocking)` — 不影响主线，可并行处理
  - 如果没有阻碍，请明确写"无"。
  - **（注意：任何报错堆栈或输出日志必须使用 ```log 或 ```csharp 代码块包裹，严禁纯文本粘贴以防破坏 Markdown 排版。）**
- **Pending Queue & Next Step (待办与下一步)**:
  - *Pending Queue*: 简要列出紧随其后的 1-2 个未完成的短期子任务。区分优先级：
    - `(P0)` — 阻塞性，必须优先处理
    - `(P1)` — 重要但非阻塞
  - *Next Step*: 明确指示新窗口的 AI 接手后，应该立即去阅读哪个文件或编写什么代码。

### 5. 错误处理

- **若文件写入失败**（权限不足、磁盘空间不足、路径不存在等）：将 Handoff Context 直接输出在对话中，并提示用户手动保存到对应 plan.md 文件。同时告知用户当前 Handoff Context 的完整内容，避免因写入失败导致状态丢失。
- **若路径根目录约定不明确**：默认以项目根目录（即当前工作目录）为基准，`docs/superpowers/plans` 为相对路径。

## 结束响应

文件保存成功后，请向用户输出以下标准化回复：

> "状态交接完毕，底层上下文已安全写入 plan 文件。
> 建议您直接关闭此窗口。在新窗口中只需发送：**『请阅读 plan 中的 Handoff Context 并继续执行』**。"
